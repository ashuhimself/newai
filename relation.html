Cluster 1: User Profile & Scoring

Central hub is tiktok_user_insights — every scoring and metric table FKs back to it. This is a classic star schema with the user as the fact table.

tiktok_user_insights
Core user table. TikTok handle, followers, bio. Central FK target for almost everything.
PK: id (uuid) 0 FKs out ~85K+ users
pulse_score
Composite quality score. Drives video classification eligibility (≥0.4 threshold).
PK: tiktok_user_insights_id FK → user_insights
engagement_score
Engagement quality rating with profile summary.
PK: tiktok_user_insights_id FK → user_insights
engagement_metrics
Detailed stats: avg views, likes, comments, saves, shares, engagement rate, quantiles.
PK: tiktok_user_insights_id FK → user_insights
breakout_score
Viral potential. Probability tiers from 1K to 100M views.
PK: tiktok_user_insights_id FK → user_insights
consistency_score
Posting consistency + relative trend + velocity metrics.
PK: tiktok_user_insights_id FK → user_insights
visibility_rate
How visible/discoverable a creator's content is.
PK: tiktok_user_insights_id FK → user_insights
expected_views
Predicted view count for future posts.
PK: tiktok_user_insights_id FK → user_insights
tiktok_post_pricing
Calculated price per sponsored post.
PK: tiktok_user_insights_id FK → user_insights
pulse_user_status
Ingestion state machine. Tracks processing status & display eligibility.
PK: tiktok_user_insights_id FK → user_insights, status_type status machine
Cluster 2: Video Classification Pipeline

Batch processing pipeline. Videos are queued, batched in groups of 100, sent to Gemini via Vertex AI, results stored and rolled up to user profiles.

batch_classification_items
Video queue. Each row = one video. Status: pending → in_batch → classified/failed/blocked.
PK: id FK → batch_classification_jobs ~284K items state machine
batch_classification_jobs
Tracks batches of ~100 videos submitted to Vertex AI. Status: preparing → submitted → completed.
PK: id (uuid) state machine
media_classifications
Per-video classification output. Categories, styles, genres (as JSON ID arrays), description, Qdrant vector_id.
PK: id ~1.56M rows
profile_description_classifications
User-level rollup. Top 3 categories, styles, genres. Created when user hits ≥9 classified videos.
PK: tiktok_user_insights_id
batch_user_classification_items
Queue for user-level classification (Stage 2). Populated when video count ≥ 9.
PK: id FK → batch_user_classification_jobs ~43K items
batch_user_classification_jobs
Tracks user-level batch jobs (Stage 2 equivalent of batch_classification_jobs).
PK: id (uuid) state machine
batch_downstream_items
Queue for re-processing users who already have a profile classification.
PK: id ~124 items
influencer_classifications
LLM-based suitability assessment. recommend, is_celebrity, content_type, gender, country, confidence.
PK: tiktok_user_insights_id FK → user_insights
Cluster 3: Music & Sounds

Sound → Song → Genre chain. Connects TikTok sounds to Spotify songs via playlists, with usage rankings per creator.

tiktok_sounds
TikTok sound entries. Links to songs, source type, processing state.
PK: sound_id FK → tiktok_songs, sound_source_type
tiktok_songs
Song metadata. Name, artist, Spotify ID, genre FK, trending flag.
PK: id (uuid) FK → genres
sound_insights
Sound metrics: UGC count, duration, is_original flag.
PK: sound_id FK → tiktok_sounds
tiktok_sound_user_ranking
Which creators rank highest for each sound. Three-way junction table.
PK: (sound_id, ranking) FK → sounds, users, media
spotify_playlists
Tracked Spotify playlists with processing state.
PK: id
playlist_songs
Junction: Spotify playlist ↔ TikTok song mapping.
PK: id (uuid) FK → spotify_playlists, tiktok_songs

Schema Observations & Issues
Missing FKs on media_classifications
Has media_id but no FK to tiktok_media_insights. The backup table has it, the main one doesn't. Also no FK to content_classifications_type or content_style_type for JSON ID arrays — validation is purely application-side.
profile_description_classifications has no FKs
PK is tiktok_user_insights_id but there's no FK constraint. Has genre_id_1/2/3 columns but no FK to genres table. The backup version has all FKs — the main table lost them during migration.
JSON columns for IDs
media_classifications stores category/style/genre IDs as JSON arrays instead of junction tables. This prevents join-based queries and makes referential integrity impossible at the DB level.
Cascade risk on user delete
Deleting one tiktok_user_insights row cascades to ~15 dependent tables. A single accidental delete could wipe scoring data, classifications, media, audience insights, pricing, and emails simultaneously.
Backup tables are drift risk
media_classifications_backup is missing the genre_ids column present in the main table. profile_description_classifications_backup is missing genre_id_1/2/3. These backups are structurally stale.
batch_downstream_items only ~124 rows
Very low usage suggests the re-classification path is barely exercised. Compared to 43K user classification items and 284K video items, this confirms the "stale data" problem — creators are rarely re-processed.
